# 测试：镍-SNBT 解析-非法输入

## 测试描述

本数据集验证 SNBT 读者（`SNBTReader.readNBTFromInput` / `readSingleNBTFromInput`）对非法输入的报错行为。

解析 API 没有返回 null / false 的失败路径：要么成功，要么经 `InputReader.panic` 抛出异常。测试环境下新建 `CommandContext`、没有命令分支上下文，所有 panic 都以 `moe.qingu.nickel.command.exception.NickelRuntimeException` 的形式抛出。因此测试只断言异常类型，不断言本地化文本——错误消息是待翻译组件，具体 i18n key 属于实现细节。

收录的非法输入分几类：

- 结构性语法错误：EOF、缺冒号、缺值、缺逗号（含复合值之后缺逗号）、空键、纯冒号、逗号开头、双逗号、函数括号未闭合、键位置的函数调用、引号字符串后接实参表；
- 函数实参表的语法错误：逗号开头、双逗号、缺逗号、实参内未闭合引号（注意实参表的尾随逗号是合法的，成功用例见 `函数调用/嵌套与语法.yaml`）；
- 复合标签重复键（duplicated_key；扫描侧不做这项检查）；
- 列表元素类型不一致；
- 字符串未闭合（含嵌套深处的未闭合）与非法转义；
- 单 NBT 入口的尾随内容（NO_SPLIT）；
- 数字字面量的各类错误：各类型溢出、非法进制位、孤立数字分隔符、前导零、无穷大、非法类型后缀、孤立正负号、孤立小数点、双小数点、浮点值配整数后缀（如 `1.5b` 的后缀错配）、指数缺失、双指数、数字与类型后缀之间的空白。

### 锁定的预期行为

下面这些用例锁定当前实现的报错方式，语义调整时应同步审视。

- `{ }` 报错：闭花括号前只有空白时，读者会先尝试读键而失败，含空白的空复合标签不被接受。这与列表不对称——`{a:[ ]}` 合法（见 `Read/`）。制表符同理，`{\t}` 也报错。
- `{a:1 b}` 报错：空白结束无引号 token，`1` 先读成 int，随后在期望闭花括号处读到 `b`。数字与类型后缀之间不允许空白，而后缀之后的空白无害（见 `Read/`）。
- `{bool(1):2}` 报错：函数调用只在值位置被识别，键读出 `bool` 后在期望冒号处读到 `(`。`{a:"bool"(1)}` 同理，引号字符串不构成函数名。
- ` {}` 报错：入口不跳过前导空白。
- `{a:天}` 报错：`天` 不在无引号字符集里，读出空 token，报"期望值"。
- `[B;300]` 报错：带类型前缀数组的元素 `300` 超出 byte 范围。
- 部分十进制整数错误（如 `128b`、`256ub`）会先落入浮点解析，再以浮点类型后缀非法而报错。测试只要求抛 `NickelRuntimeException`，不锁定具体错误种类。

## 样例格式

YAML 用例列表，每个用例的键：

- `name`：用例名；
- `input`：SNBT 输入原文（YAML 单引号书写）；
- `mode`：可选，取 `single` 时经 `readSingleNBTFromInput` 入口。

没有 `expected` 键：所有用例统一断言解析抛出 `NickelRuntimeException`。

## 样例 #1

### 输入 #1

```yaml
- name: 缺失值
  input: '{a:}'
```

## 说明/提示

### 样例 1 解释

键 `a` 后的冒号之后直接闭合，读值时得到空 token，读者 panic"期望一个值"。测试断言抛出 `NickelRuntimeException`。
